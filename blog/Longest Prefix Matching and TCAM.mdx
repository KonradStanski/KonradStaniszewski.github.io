Every time you send a packet across the internet, dozens of routers must each decide: *where does this packet go next?* That decision, made billions of times per second across the internet, relies on two ideas — **longest prefix matching** and a specialized piece of hardware called **TCAM**. Let's break down how they work.

## IP Addresses in Binary

Before understanding route lookup, it helps to see what an IP address looks like under the hood. An IPv4 address is a 32-bit number, typically written as four decimal octets.

<BinaryBreakdown />

Routers don't think in dotted decimal — they operate on these raw bits. A route like `10.1.1.0/24` means "match the first 24 bits." The `/24` is the **prefix length**, and it tells the router exactly how many leading bits matter.

## What Is Prefix Matching?

A router's forwarding table is a list of prefixes, each pointing to a next-hop destination. When a packet arrives, the router compares the destination IP against every prefix in the table.

Multiple prefixes can match the same IP. Consider a packet destined for `10.1.1.200`:

- `0.0.0.0/0` matches (it matches everything — it's the default route)
- `10.0.0.0/8` matches (first 8 bits agree)
- `10.1.0.0/16` matches (first 16 bits agree)
- `10.1.1.0/24` matches (first 24 bits agree)
- `10.1.1.128/25` matches (first 25 bits agree — `200` = `11001000`, and the top bit is `1`)

### Longest Prefix Match Wins

The router always picks the **most specific** matching route — the one with the longest prefix. This is the **longest prefix match** (LPM) rule. A longer prefix means a more specific route, which presumably has better information about how to reach that particular subnet.

## The Problem: Speed

A modern core router handles 10-100+ Gbps of traffic. With 64-byte minimum-size packets, that's potentially **hundreds of millions of packets per second**, each needing a route lookup. A full internet routing table has over **900,000 prefixes**.

A naive approach — linearly scanning the table — would take O(N) comparisons per packet. Even a trie (the standard software data structure for prefix matching) requires O(W) memory accesses, where W=32 for IPv4. With DRAM latency around 10 ns per access, that's ~320 ns per lookup — far too slow for line-rate forwarding.

## Enter TCAM: Ternary Content-Addressable Memory

**TCAM** (Ternary Content-Addressable Memory) is a specialized memory chip that solves this problem through parallelism.

### How Regular Memory Works

In conventional RAM, you provide an **address** and get back the **data** stored there. You search for content by reading addresses one at a time.

### How TCAM Works

TCAM flips this around. You provide the **content** (the destination IP) and the hardware tells you **where it matches** — checking every single entry **simultaneously in one clock cycle**.

The "ternary" part means each bit in a TCAM entry can be one of three states:
- **0** — must match a zero
- **1** — must match a one
- **X** — don't care (matches either)

This is perfect for prefix matching. A `/24` route stores 24 specific bits followed by 8 don't-care bits. A `/16` stores 16 specific bits and 16 don't-care bits.

### How It's Organized for LPM

The TCAM entries are ordered by prefix length, longest first. When multiple entries match, a **priority encoder** circuit selects the first match (highest priority = longest prefix). This gives us longest prefix match in a single hardware operation.

The key insight: *all comparisons happen in parallel*. Whether the table has 100 entries or 1,000,000, the lookup takes the same amount of time — one TCAM cycle.

### Inside a TCAM Cell

Each TCAM cell is built from a pair of SRAM cells storing two bits, which together encode the three possible states:

| Stored Bit 1 | Stored Bit 2 | Meaning |
|:---:|:---:|:---|
| 0 | 0 | Match 0 only |
| 1 | 1 | Match 1 only |
| 0 | 1 | Don't care (X) |

When a search key is broadcast across the TCAM array, each cell independently compares its stored value against the search bit. A match line runs across each row — if *every* cell in a row matches, that row's match line stays high. The priority encoder then picks the highest-priority (top-most) match.

This is fundamentally different from RAM. There are no sequential memory accesses, no pointer chasing, no tree traversal. Every entry is compared at once via dedicated comparison circuitry built into each memory cell.

### The Tradeoffs

TCAM is not free. It comes with significant costs:

| | TCAM | SRAM/DRAM (software) |
|:---|:---|:---|
| **Lookup speed** | O(1) — single cycle | O(W) — 32 accesses worst case |
| **Power consumption** | Very high (every cell active on every lookup) | Low (only accessed cells active) |
| **Cost per bit** | ~10-20x more than SRAM | Baseline |
| **Density** | Low (16 transistors/cell) | High (6 transistors/cell for SRAM) |
| **Heat** | Significant cooling required | Minimal |
| **Capacity** | Typically 1-20M entries | Virtually unlimited |

The power cost is substantial — every lookup activates every cell in the TCAM simultaneously. This is why high-end routers consume kilowatts of power and need active cooling. It's also why TCAM capacity is always limited: more entries means proportionally more power and heat.

### Why Not Just Use Faster Software?

Modern CPUs are fast, but the physics are against them for this task. A trie lookup requires multiple *dependent* memory accesses — you can't fetch the next node until you've read the current one. CPU caches help, but a full internet routing table doesn't fit in L1/L2 cache, so most accesses go to slower DRAM.

TCAM bypasses this entirely. There's no pointer chasing, no cache misses, no branch prediction. It's brute-force parallelism implemented in silicon.

## Beyond IPv4

The same TCAM principles extend to:

- **IPv6** — same concept but 128-bit addresses; requires more TCAM space per entry
- **ACLs** (Access Control Lists) — match on source IP, destination IP, port, protocol simultaneously
- **MPLS label lookups** — exact match on 20-bit labels
- **Policy-based routing** — match on arbitrary header field combinations

In each case, the ternary match capability and parallel comparison make TCAM the hardware of choice for classification tasks that must run at line rate.

## Bonus: Prefix Compression — Squeezing More Out of TCAM

TCAM is expensive, power-hungry, and limited in capacity. Every entry you can eliminate saves silicon, watts, and heat. This is where **prefix compression** (also called route aggregation or supernetting) comes in.

### The Core Idea

If two adjacent prefixes of the same length point to the same next hop, they can be merged into a single shorter prefix that covers both. This is the inverse of subnetting — instead of splitting a network into smaller pieces, we're combining smaller pieces back into a larger one.

Consider these two /24 routes:

- `192.168.0.0/24` → Router A (binary prefix: `11000000.10101000.00000000`)
- `192.168.1.0/24` → Router A (binary prefix: `11000000.10101000.00000001`)

These two prefixes are identical in their first 23 bits and differ only in bit 24. Since they share the same next hop, we can replace both with a single `/23`:

- `192.168.0.0/23` → Router A (binary prefix: `11000000.10101000.0000000`)

One TCAM entry instead of two, with identical forwarding behavior.

### When Can Routes Be Compressed?

Two routes can merge into a parent when all three conditions hold:

1. **Same prefix length** — both are `/N`
2. **Same next hop** — packets from both go to the same place
3. **Sibling prefixes** — they share the first `N-1` bits and differ only at bit `N` (one has a `0`, the other a `1`)

This process is recursive. After merging two `/24`s into a `/23`, that `/23` might be mergeable with an adjacent `/23` into a `/22`, and so on.

### Why This Matters at Scale

The full internet routing table has over 900,000 prefixes. Without compression, each one occupies a TCAM row. ISPs and router vendors use aggressive aggregation strategies:

- **Route summarization at borders** — an ISP advertises a single aggregate for its entire address space instead of thousands of individual customer routes
- **Optimal TCAM encoding** — dedicated algorithms (like the Espresso logic minimizer) find the smallest set of ternary entries that produce identical forwarding behavior
- **Dynamic re-aggregation** — some routers continuously re-optimize their TCAM contents as routes are added and withdrawn

Even modest compression — reducing the table by 10-20% — can mean the difference between fitting in TCAM or needing a more expensive chip. Given that TCAM costs ~10-20x more per bit than SRAM, every saved entry has real dollar value.

<Note>
Route compression must be done carefully. Over-aggregation can create "black holes" — advertising reachability for address space you can't actually reach. This is why compression only merges routes with the *same* next hop: the forwarding behavior is guaranteed to be identical.
</Note>

## Summary

Longest prefix matching is what lets the internet scale — hierarchical addressing means routers only need to know about network prefixes, not individual hosts. TCAM is the hardware that makes this lookup fast enough to forward every packet at wire speed. The combination of a well-defined algorithmic rule (longest prefix match) and purpose-built silicon (TCAM with priority encoding) is what allows a single router to handle billions of forwarding decisions per second.
